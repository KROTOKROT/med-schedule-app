# 01. Requirements (Модель требований к системе)

## 1. Общая информация

Система: MedSchedule — мобильное приложение для управления расписанием приёма лекарств.

Цель этапа — формализация системных требований на основе бизнес-модели (этап 0).

---

## 2. Пользователи системы (акторы)

- Пользователь (Patient) — основной пользователь приложения
- Система уведомлений Android — доставка напоминаний
- Сервер (Backend) — хранение и синхронизация данных

---

## 3. Use Case диаграмма (описание)

Основные сценарии использования системы:

### Пользователь:
- Создать расписание приёма лекарства
- Добавить лекарство
- Редактировать расписание
- Удалить лекарство
- Просмотреть список лекарств
- Отметить приём лекарства
- Просмотреть историю приёма

### Система:
- Отправить уведомление о приёме
- Синхронизировать данные с сервером
- Сохранить данные локально (offline mode)

---

## 4. Domain Model (модель предметной области)

### Основные сущности:

**User**
- id
- name
- email
- password

**Medication**
- id
- name
- dosage
- description

**Schedule**
- id
- medicationId
- time
- frequency
- startDate
- endDate

**IntakeRecord**
- id
- scheduleId
- takenAt
- status (taken / missed)

---

## 5. Связи между сущностями

- User → имеет множество Medication
- Medication → имеет множество Schedule
- Schedule → имеет множество IntakeRecord

---

## 6. Основные Use Cases

### UC-1: Создание лекарства
Пользователь добавляет новое лекарство в систему.

Результат:
- лекарство сохраняется в локальной БД и сервере

---

### UC-2: Создание расписания
Пользователь задаёт время и частоту приёма лекарства.

Результат:
- создаётся schedule
- формируются уведомления

---

### UC-3: Отметка приёма
Пользователь отмечает, что принял лекарство.

Результат:
- создаётся IntakeRecord
- обновляется история

---

### UC-4: Получение уведомления
Система отправляет напоминание в заданное время.

Результат:
- push/local notification

---

## 7. Ограничения системы

- Работа в офлайн-режиме обязательна
- Данные должны синхронизироваться при появлении сети
- Авторизация через JWT
- Минимум 5 экранов в мобильном приложении

---

## 8. Вывод

На этапе требований определены основные сущности системы, сценарии использования и ограничения. Это формирует основу для архитектурного проектирования (PCMEF).
