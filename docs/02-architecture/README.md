# 02. Architecture (Архитектурное проектирование)

## 1. Общая архитектура системы

Система MedSchedule построена на архитектурном паттерне **PCMEF (Presentation-Control-Mediator-Entity-Foundation)** и реализует клиент-серверную модель.

---

## 2. Общая схема системы

### Клиент (Mobile - Flutter)
- Presentation: экраны приложения
- Control: ViewModel / state management
- Mediator: Repository слой
- Entity: доменные модели
- Foundation: API + локальная БД (Room/SQLite аналог)

### Сервер (Spring Boot)
- Presentation: REST Controllers
- Control: DTO + validation layer
- Mediator: Service layer (business logic)
- Entity: JPA entities
- Foundation: Repository (Spring Data JPA)

---

## 3. PCMEF архитектура

### Принцип:
Зависимости направлены строго сверху вниз:

Presentation → Control → Mediator → Entity → Foundation

---

## 4. Слои архитектуры

| Слой | Назначение |
|------|-----------|
| Presentation | UI (Flutter экраны) |
| Control | Управление состоянием, обработка действий |
| Mediator | Бизнес-логика приложения |
| Entity | Доменные модели |
| Foundation | Доступ к данным (API + БД) |

---

## 5. Взаимодействие клиента и сервера

- Мобильное приложение работает через REST API
- Передача данных: JSON
- Авторизация: JWT токены
- Синхронизация: при появлении сети

---

## 6. Основные архитектурные решения (ADR)

### ADR-001: Выбор архитектуры PCMEF
**Решение:** Использовать PCMEF как основной архитектурный паттерн  
**Причина:** Требование курсовой и необходимость строгого разделения слоёв  
**Последствия:** Упрощение тестирования и масштабирования

---

### ADR-002: Выбор мобильной технологии
**Решение:** Flutter  
**Причина:** Кроссплатформенность и единая кодовая база  
**Альтернатива:** Native Android (Kotlin)  
**Последствия:** Ускорение разработки, но зависимость от Flutter SDK

---

### ADR-003: Выбор backend
**Решение:** Spring Boot (Java 17)  
**Причина:** Требование курсовой + зрелая экосистема  
**Последствия:** Надёжный backend с поддержкой REST и безопасности

---

### ADR-004: Хранение данных
**Решение:** PostgreSQL + локальная SQLite  
**Причина:** Серверное хранение + офлайн режим  
**Последствия:** Необходимость синхронизации данных

---

## 7. Безопасность

- JWT авторизация
- BCrypt хеширование паролей
- Разделение ролей (USER / ADMIN)
- Защищённые REST endpoints

---

## 8. Вывод

Архитектура системы обеспечивает:
- масштабируемость
- разделение ответственности (PCMEF)
- поддержку офлайн-режима
- безопасное взаимодействие клиента и сервера
