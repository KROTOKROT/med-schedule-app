# 04. Detailed Design (Детальное проектирование)

## 1. Общая информация

На данном этапе описаны ключевые сценарии работы системы MedSchedule в виде последовательностей взаимодействий между компонентами PCMEF-архитектуры.

---

## 2. Сценарий UC-1: Создание лекарства

### Описание:
Пользователь добавляет новое лекарство в систему.

### Участники:
- User (клиент)
- Presentation (UI)
- Control (ViewModel / Controller)
- Mediator (Service)
- Foundation (Repository / DB)

### Последовательность:

1. Пользователь вводит данные лекарства в UI
2. Presentation передаёт данные в Control слой
3. Control выполняет валидацию
4. Mediator обрабатывает бизнес-логику
5. Foundation сохраняет данные в БД
6. Система возвращает результат в UI

---

## 3. Сценарий UC-2: Создание расписания

### Описание:
Пользователь задаёт время приёма лекарства.

### Последовательность:

1. Пользователь выбирает лекарство
2. Вводит время и частоту приёма
3. Control валидирует данные
4. Mediator создаёт Schedule
5. Foundation сохраняет в PostgreSQL
6. UI обновляет список расписаний

---

## 4. Сценарий UC-3: Отметка приёма лекарства

### Описание:
Пользователь отмечает факт приёма лекарства.

### Последовательность:

1. Пользователь нажимает “Принято”
2. UI отправляет событие в Control
3. Mediator создаёт IntakeRecord
4. Foundation сохраняет запись
5. UI обновляет историю

---

## 5. Сценарий UC-4: Получение уведомления

### Описание:
Система отправляет напоминание о приёме.

### Последовательность:

1. Scheduler на сервере проверяет расписание
2. Генерируется событие уведомления
3. Mobile OS получает push/local notification
4. Пользователь видит напоминание

## 7. Взаимодействие слоёв PCMEF

- Presentation → Control: UI события
- Control → Mediator: бизнес-запросы
- Mediator → Foundation: работа с БД
- Foundation → Database: SQL операции
